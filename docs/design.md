# NbReader 设计文档

## 1. 文档目的

本文档是 NbReader 的正式设计文档，用于统一产品目标、范围边界、核心架构和首版实现策略。

NbReader 的目标是构建一款面向重度漫画读者的跨平台阅读软件，优先解决以下问题：

1. 本地漫画资源混乱，难以整理和检索
2. zip 和图片目录两种常见来源体验不统一
3. 系列、卷号、阅读进度不可靠
4. 漫画阅读器常见操作成本高，翻页体验不稳定
5. 在线来源能力容易反向拖垮整个项目节奏

本设计文档采用以下技术路线：

1. Avalonia + .NET + SQLite
2. 首版优先支持 zip 与图片文件夹
3. 本地漫画库管理和稳定阅读优先于在线来源

配套设计文档：

1. [database-entities-and-fields.md](database-entities-and-fields.md)
2. [import-flow-and-error-state-machine.md](import-flow-and-error-state-machine.md)
3. [development-plan.md](development-plan.md)

## 2. 产品定位

NbReader 是一款以本地漫画库为核心的跨平台漫画阅读器，面向同时重视“收藏整理”和“阅读体验”的用户。

### 2.1 核心定位

1. 它不是单纯的图片查看器
2. 它不是以在线爬取为核心的采集工具
3. 它是一个围绕系列管理、稳定导入、快速检索和低干扰阅读构建的漫画书库系统

### 2.2 目标平台

1. Windows
2. Linux
3. macOS

### 2.3 目标用户

1. 本地存有大量漫画资源的重度读者
2. 习惯按系列、作者、题材管理漫画的收藏型用户
3. 需要同时阅读 zip 与图片文件夹资源的用户
4. 后续可能需要接入在线来源，但首版更看重离线稳定性的用户

## 3. 设计原则

### 3.1 图书馆思维优先于文件夹思维

物理文件结构不可信，数据库中的元数据索引才是产品中的真相来源。

### 3.2 系列优先，但保留物理来源视角

对用户而言，漫画首先是一个 Series，其次才是一个个 Volume；但在实现上，必须保留 Source 这一物理来源层，才能处理混乱命名、重复资源、远程导入和重扫。

### 3.3 阅读器与资料库是两条不同任务流

1. 资料库强调筛选、排序、归类和批量整理
2. 阅读器强调低延迟、低干扰、低操作成本

因此两者共享数据，但不应强行共享交互模式。

### 3.4 自动识别要克制

真实漫画资源往往存在命名混乱、目录结构异常、编码不统一和多版本混杂的问题。系统应该尽量自动识别，但在不确定时必须允许用户确认和修正。

### 3.5 首版先做“可靠”，再做“聪明”

首版必须先保证：

1. 导入可靠
2. 系列和卷号可信
3. 搜索可用
4. 阅读顺手

智能裁边、在线元数据匹配、插件系统都应后置。

## 4. 产品范围

### 4.1 首版必须支持

1. 漫画资料库浏览
2. zip 导入
3. 图片文件夹导入
4. 系列识别与卷排序
5. 作者、年份、标签、状态等维度检索
6. 单页和双页阅读
7. 右到左与左到右阅读方向切换
8. 阅读进度持久化
9. 导入任务视图与错误提示
10. 基础封面与缩略图缓存

### 4.2 首版明确不做

1. rar 和 7z 正式支持
2. 内建侵权在线源
3. 智能裁边
4. 跨设备同步
5. 手机端配套
6. 页面批注和书签系统
7. 高级推荐与内容分析

### 4.3 后续扩展方向

1. Provider 形式的 URL 导入
2. 在线元数据匹配
3. 系列合并、拆分、卷号修正工具
4. 同步服务
5. 插件系统

## 5. 核心用户场景

### 5.1 本地书库整理

用户拖入单个 zip、单个图片目录或一个系列目录。系统自动识别系列、卷号、页面顺序，必要时让用户确认，并把结果写入书库。

### 5.2 系列浏览与继续阅读

用户在主界面按系列浏览书架，通过最近阅读、继续阅读、作者或标签快速定位目标系列，再进入具体卷继续阅读。

### 5.3 沉浸式阅读

用户以单页或双页模式阅读漫画，系统自动保留页码、阅读方向、缩放模式和跨卷继续阅读状态。

### 5.4 问题资源修正

当目录结构混乱、卷号识别不准或出现重复卷时，用户可以在导入确认阶段或导入后手动修正系列归属和卷排序。

### 5.5 URL 导入扩展

后续用户可以通过 URL 导入远程作品，但首版产品架构只预留能力，不把它纳入交付主目标。

## 6. 信息架构

### 6.1 核心概念模型

产品内部采用三层视角：

1. Source：物理来源，可能是 zip、目录或远程缓存
2. Volume：用户可以直接阅读的一本卷册
3. Series：书架中的聚合视图

### 6.2 主要导航结构

建议首版主导航包含：

1. 全部漫画
2. 继续阅读
3. 最近阅读
4. 最近添加
5. 未整理
6. 导入失败
7. 标签
8. 作者
9. 设置

### 6.3 主界面展示原则

1. 以系列卡片为主
2. 系列默认聚合显示卷数和阅读进度
3. 点击系列优先进入详情视图，而不是直接打开第一卷
4. 支持网格视图和列表视图

### 6.4 阅读界面展示原则

1. 阅读区域优先级最高
2. 非必要控件默认隐藏
3. 缩略图导航和进度条按需出现
4. 阅读配置应支持全局默认值和系列级覆盖

---

## 7. 功能设计

### 7.1 导入系统

导入系统必须任务化，不能只实现为一次性扫描函数。

首版导入支持：

1. 单个 zip
2. 单个图片目录
3. 系列目录
4. 多路径批量导入

导入流程要求：

1. 自动识别结构
2. 生成页面清单
3. 猜测系列名和卷号
4. 检测冲突和重复
5. 在不确定时进入用户确认阶段
6. 将主体数据写入数据库
7. 后台生成封面、缩略图和索引

详细流程见 [import-flow-and-error-state-machine.md](import-flow-and-error-state-machine.md)。

### 7.2 书库管理

书库系统需要支持：

1. 系列聚合视图
2. 卷级详情视图
3. 最近添加与最近阅读
4. 未整理与导入失败视图
5. 手动修正系列、卷号和部分元数据

### 7.3 搜索与筛选

首版搜索必须同时支持：

1. 标题搜索
2. 作者搜索
3. 标签筛选
4. 年份筛选
5. 状态筛选
6. 已读与未读筛选

检索设计要求：

1. 结构化字段使用数据库字段过滤
2. 模糊搜索使用 FTS 或专用索引
3. 拼音搜索通过额外索引字段支持
4. 标签需要按类别管理，而不是扁平字符串集合

### 7.4 阅读器

首版阅读器支持：

1. 单页模式
2. 双页模式
3. 右到左与左到右阅读方向
4. 适应宽度、适应高度、原始比例
5. 当前页前后若干页预加载
6. 具体页码进度保存
7. 跨卷继续阅读

阅读体验要求：

1. 双页模式下封面单独显示
2. 不使用百分比进度替代页码
3. 避免整卷预解码导致内存暴涨
4. 退出、失焦和翻页过程中要有明确的进度写回策略

### 7.5 URL 导入预留

首版不交付完整 URL 导入，但架构上预留 Provider 接口，以保证未来扩展时不破坏现有导入与阅读抽象。

## 8. 技术架构

### 8.1 技术选型

1. UI：Avalonia
2. 运行时：.NET
3. 数据库：SQLite
4. 图像解码与渲染：SkiaSharp 为主，ImageSharp 作为离线处理补充
5. 压缩包处理：SharpCompress，首版只正式承诺 zip

### 8.2 模块划分

建议代码结构至少分为以下模块：

1. Catalog
2. Import
3. Reader
4. Metadata
5. Search
6. Infrastructure

### 8.3 线程与后台任务策略

1. UI 线程只负责展示与用户交互
2. 文件扫描、页面分析、缩略图生成、索引构建在后台执行
3. 导入和后处理必须可中断、可重试、可记录事件

### 8.4 页面读取抽象

阅读器页源需要统一抽象，以同时支持 folder、zip 和 future remote/cache：

1. OpenVolume
2. EnumeratePages
3. ReadPageStream
4. GetPageMetadata

### 8.5 缓存策略

1. 内存层使用 LRU 缓存热页和热封面
2. 磁盘层缓存封面和缩略图
3. 不把原始页面图像存进数据库
4. zip 只做流式读取和页索引缓存，不解压整卷到临时目录

## 9. 数据设计

正式字段和表结构定义见 [database-entities-and-fields.md](database-entities-and-fields.md)。

首版数据库主骨架为：

1. Series
2. Volume
3. Source
4. Page
5. Person
6. Tag
7. ReadingProgress
8. ImportTask
9. Asset

数据设计原则：

1. 数据库是元数据真相来源
2. Source 承担幂等导入和重扫能力
3. Volume 是首版书架原子单元
4. Series 是用户视角聚合层
5. 搜索字段要提前规范化

## 10. 非功能要求

### 10.1 性能

1. 常规卷翻页应接近无等待
2. 书架首屏应优先展示已缓存封面
3. 导入大批量资源时 UI 不应阻塞
4. 大图资源不可整卷预解码

### 10.2 可靠性

1. 导入必须幂等
2. 导入失败必须可诊断
3. 源文件丢失后不应立即破坏书库记录
4. 关键主体数据与后处理结果应分离提交

### 10.3 跨平台

1. 正确处理非 ASCII 文件名
2. 处理路径长度与大小写敏感差异
3. 处理平台文件锁行为差异
4. 遵守 Linux XDG 目录规范

### 10.4 可维护性

1. UI、导入、阅读、搜索、元数据模块边界清晰
2. 后续引入 URL provider 时不推翻现有模型
3. 不过早为 rar/7z、同步、插件做过度设计

---

## 11. 风险与约束

### 11.1 高风险点

1. 混乱目录结构导致系列和卷识别错误
2. zip 内文件名编码异常
3. 超大图片带来的内存压力
4. URL 导入后续可能带来的法务与维护成本
5. 过早支持过多格式造成异常处理面失控

### 11.2 应对策略

1. 导入阶段引入 awaiting_confirmation
2. 通过 Source 和 ImportTask 保证幂等与可恢复
3. 通过统一页源抽象隔离来源差异
4. 首版只承诺 zip 和图片文件夹
5. 先做好本地阅读器和书库，再扩在线来源

---

## 12. 版本策略

### 12.1 MVP 目标

MVP 的完成标准不是“功能看起来很多”，而是形成一个稳定闭环：

1. 导入
2. 整理
3. 检索
4. 阅读
5. 持久化

### 12.2 首版验收标准

1. 用户可稳定导入 zip 和图片目录
2. 用户可在多数常见场景下正确看到系列和卷顺序
3. 用户可按标题、作者、tag 和年份找到漫画
4. 用户可顺畅阅读并保留进度
5. 用户可理解导入失败原因并进行修正

### 12.3 扩展原则

所有增强功能都必须建立在 MVP 稳定的前提下，再按以下顺序推进：

1. 系列修正能力增强
2. 搜索与元数据增强
3. URL provider
4. 同步与插件化

---

## 13. 结论

NbReader 的首版不应追求“功能列表最大化”，而应专注于一条清晰的产品主线：

构建一个以本地漫画库为核心、系列管理可靠、阅读体验稳定、后续可演进的跨平台漫画阅读器。

基于这个目标，正式技术路线确定为：

1. Avalonia + .NET + SQLite
2. zip 与图片文件夹优先
3. 导入任务化
4. 阅读器页源抽象优先定义
5. 系列整理能力视为核心功能

只要这条主线保持清晰，后续扩展 URL 导入、在线元数据和插件系统时，系统不会失控。
