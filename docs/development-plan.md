# NbReader 渐进式开发计划

## 1. 计划目标

本文档定义 NbReader 从正式设计走向可交付产品的渐进式开发路径。

计划目标：

1. 先打通最小可用闭环
2. 在每一阶段结束时都保留可运行、可验证的产品状态
3. 优先解决导入、检索、阅读这三条主链路
4. 延后高复杂度和高维护成本能力

本计划与以下文档配套使用：

1. [design.md](design.md)
2. [database-entities-and-fields.md](database-entities-and-fields.md)
3. [import-flow-and-error-state-machine.md](import-flow-and-error-state-machine.md)

---

## 2. 总体策略

### 2.1 开发原则

1. 每个阶段只解决少数关键问题
2. 每个阶段结束都必须能演示真实用户场景
3. 所有新增能力都优先复用既有数据模型和导入抽象
4. 避免在首版前同时展开 UI 美化、在线源、同步和插件系统

### 2.2 实施节奏

建议采用以下节奏：

1. Phase 0：项目起盘与技术验证
2. Phase 1：本地导入闭环
3. Phase 2：书架与阅读闭环
4. Phase 3：搜索与整理闭环
5. Phase 4：MVP 打磨与发布准备
6. Phase 5：增强功能

### 2.3 成功标准

如果某一阶段结束后仍无法稳定演示核心场景，就不应进入下一阶段。

---

## 3. Phase 0：项目起盘与技术验证

### 3.1 目标

验证正式技术路线可行，并建立后续开发骨架。

### 3.2 关键产出

1. Avalonia 应用基础骨架
2. 项目分层和目录结构
3. SQLite 基础接入
4. 日志和配置基础设施
5. 最小图片显示能力
6. zip 页面枚举验证

### 3.3 需要验证的问题

1. Avalonia 在三平台上的基础运行情况
2. SkiaSharp 渲染是否满足阅读器需求
3. SharpCompress 对 zip 漫画页枚举是否稳定
4. SQLite 与 FTS5 可否按预期使用
5. 非 ASCII 路径读取是否稳定

### 3.4 交付标准

1. 应用可启动
2. 可打开一个本地图片目录并显示第一页
3. 可读取一个 zip 并枚举内部图片
4. 可创建数据库并写入测试数据

### 3.5 风险

1. Avalonia 与图像库组合的兼容问题
2. zip 内编码异常导致读取失败
3. 路径编码和文件锁问题在 Windows/macOS/Linux 行为不一致

---

## 4. Phase 1：本地导入闭环

### 4.1 目标

打通“输入路径 -> 识别 -> 确认 -> 入库”的导入主链路。

### 4.2 范围

1. 单个 zip 导入
2. 单个图片目录导入
3. 系列目录导入
4. ImportTask 与 ImportTaskEvent 基础实现
5. Source、Series、Volume、Page 的基础入库
6. 导入冲突检测
7. 用户确认阶段

### 4.3 关键任务

1. 实现路径标准化与幂等识别
2. 实现文件和目录结构识别
3. 实现有效图片筛选与自然排序
4. 实现系列名和卷号候选提取
5. 实现 ImportPlan
6. 实现 awaiting_confirmation 流程
7. 实现主体数据事务写入

### 4.4 验收场景

1. 导入单个 zip 后能在数据库中形成完整 Volume + Page 记录
2. 导入系列目录后能正确生成多个卷
3. 遇到混合目录时能要求用户确认
4. 重复导入同一路径时不会产生重复记录
5. 导入失败时能看到明确错误原因

### 4.5 不做项

1. 批量 URL 导入
2. 高级元数据匹配
3. 完整缩略图系统

---

## 5. Phase 2：书架与阅读闭环

### 5.1 目标

打通“资料库浏览 -> 进入卷 -> 阅读 -> 保存进度”的阅读闭环。

### 5.2 范围

1. 系列卡片视图
2. 系列详情视图
3. 卷列表和卷打开
4. 单页模式
5. 双页模式
6. 阅读方向切换
7. 基础预加载
8. ReadingProgress 持久化
9. 最近阅读与继续阅读

### 5.3 关键任务

1. 实现主界面导航框架
2. 实现系列聚合查询
3. 实现卷详情页和打开逻辑
4. 实现统一页源抽象
5. 实现 ReaderState 和进度写回策略
6. 实现封面单页和双页配对规则

### 5.4 验收场景

1. 用户能从系列进入卷并稳定翻页
2. 关闭应用后能恢复到上次页码
3. 双页模式方向切换正确
4. 当前卷结束后能进入下一卷
5. 最近阅读和继续阅读视图可用

### 5.5 风险

1. 双页配对逻辑复杂度被低估
2. 大图预加载策略不当导致内存问题
3. 进度写回过于频繁导致数据库压力或 UI 卡顿

---

## 6. Phase 3：搜索与整理闭环

### 6.1 目标

让书库真正可管理、可检索，而不是只能看封面。

### 6.2 范围

1. 标题搜索
2. 作者搜索
3. 标签筛选
4. 年份和状态筛选
5. 未整理视图
6. 导入失败视图
7. 基础系列修正能力
8. 基础手动元数据编辑

### 6.3 关键任务

1. 实现 Person、Tag 及关联表
2. 建立 FTS 和拼音索引字段
3. 实现结构化过滤条件
4. 实现系列合并、改名和卷号修正最小能力
5. 实现未整理与导入失败数据视图

### 6.4 验收场景

1. 能按标题、作者、tag、年份定位漫画
2. 混乱导入结果可通过 UI 修正
3. 导入失败记录可被用户重新处理
4. 系列聚合在修正后立即生效

### 6.5 不做项

1. 高级搜索语法完整实现
2. 在线元数据自动匹配
3. 自动智能标签批量落库

---

## 7. Phase 4：MVP 打磨与发布准备

### 7.1 目标

将已有闭环打磨到可作为正式 MVP 发布的水平。

### 7.2 范围

1. 缩略图缓存系统
2. 导入性能优化
3. 阅读器性能优化
4. 基础设置页
5. 错误恢复与日志完善
6. 三平台打包验证
7. 文档与发布说明

### 7.3 关键任务

1. 完成封面和缩略图后台生成
2. 完成导入任务列表体验打磨
3. 优化首次打开和切卷体验
4. 补充异常路径测试
5. 打通 Windows、Linux、macOS 三平台构建
6. 输出基础使用文档

### 7.4 MVP 验收标准

1. zip 和图片目录导入稳定
2. 系列与卷排序在常见场景正确
3. 搜索与筛选能覆盖核心管理需求
4. 阅读器能稳定保存进度并快速翻页
5. 导入错误可理解、可重试、可修正
6. 三平台均能启动并完成核心流程

### 9.1 建议的代码模块拆分

1. AppShell
2. Catalog
3. Import
4. Reader
5. Search
6. Metadata
7. Infrastructure

### 9.2 建议的实施顺序

1. Infrastructure
2. Import 基础链路
3. Catalog 基础视图
4. Reader 基础能力
5. Search 与 Metadata
6. Import 增强和打磨

### 9.3 每周节奏建议

如果按单人或小团队节奏推进，建议每周都保留以下固定结构：

1. 一个核心功能目标
2. 一组回归验证场景
3. 一次异常路径清理
4. 一次文档同步

## 10. 测试与验证计划

### 10.1 核心测试样本库

建议尽早建立一套真实测试资源，用于持续回归：

1. 单卷 zip
2. 单卷图片目录
3. 系列二级目录
4. 根目录混合图片和子目录
5. 带日文文件名 zip
6. 大尺寸扫描页资源
7. 重复卷和重复系列样本

### 10.2 核心回归场景

1. 重复导入同一路径
2. 导入过程中取消任务
3. 源文件缺失后重新打开书架
4. 双页模式阅读封面页
5. 从最近阅读恢复进度
6. 修正系列后检索结果更新

### 10.3 发布前验证

1. 三平台启动验证
2. 数据库初始化验证
3. 导入性能抽样验证
4. 阅读器长时间运行稳定性验证
5. 非 ASCII 路径与文件名验证

## 11. 里程碑建议

### 里程碑 A

项目可以读取 zip 和目录，并把卷和页面写入数据库。

### 里程碑 B

项目可以展示系列卡片、打开卷并保留阅读进度。

### 里程碑 C

项目可以搜索、筛选、修正部分错误导入结果。

### 里程碑 D

项目达到 MVP 发布标准。

建议只有在当前里程碑可稳定演示后，才进入下一个里程碑。

---

## 12. 最终建议

NbReader 的开发不能按“先堆功能，后补结构”的方式推进。更稳的路径是：

1. 先把本地导入做稳
2. 再把阅读闭环做顺
3. 再把检索和整理做强
4. 最后再碰在线来源和插件化

这条路径的优点是每一阶段都有明确产出，而且不会因为早期引入高复杂度能力把整个项目拖偏。
