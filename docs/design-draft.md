## 一、核心设计理念

### 1. 图书馆思维而非文件夹思维
不要强迫用户按文件夹结构管理，而是建立**元数据层**。物理文件可以乱糟糟，但图书馆目录必须井然有序。

补充建议：元数据层不要只当作“展示层”，而要成为整个产品的真相来源。用户看到的系列、卷、作者、标签、阅读进度，应该全部来自数据库索引，而不是每次临时扫描文件系统现算。

### 2. 渐进式披露
- 主界面只展示封面（视觉记忆比文字快10倍）
- 详细信息按需展开
- 阅读界面零干扰（鼠标不动时全隐藏）

补充建议：阅读器和资料库是两种不同任务流。资料库界面强调筛选、排序、批量整理；阅读界面强调沉浸、低延迟、低操作成本。两者不应共用一套过于复杂的交互。

### 3. 系列优先于单本
漫画是连续叙事，**"鬼灭之刃"是一个系列，包含23卷+外传**，绝不能分散显示。

补充建议：系列优先是对的，但要保留“单本视角”和“物理文件视角”。因为很多用户的本地资源并不干净：
- 有些系列会混入番外、设定集、短篇集
- 有些压缩包本身就是单话而非单卷
- 同一系列可能来自多个来源，命名并不统一

所以产品内部建议区分三层：
- **Source Item**：一个物理导入项，可能是 zip、目录、远程章节
- **Book/Volume**：用户认知中的一本
- **Series**：书架中的聚合视图

这三层分开后，后续做重命名、合并系列、拆分卷册、去重会轻松很多。

## 二、Review结论

这份设计稿的方向是正确的，尤其是以下几点：

1. 以系列为核心，而不是以文件夹为核心
2. 把阅读体验放在和管理体验同等重要的位置
3. 预判了大图、压缩包、缩略图缓存这些真正会影响体验的问题

但从落地角度看，还缺四个关键决策：

1. **导入边界不够清晰**
    本地导入和 URL 导入实际上是两条不同链路，数据可信度、失败模式、缓存策略都不同，应该拆开设计。
2. **数据模型还偏展示化**
    当前模型有 Series、Volume、Page，但缺少 Source、Chapter、Person、TagLink、ReadingHistory 这些后续必然要补的实体。
3. **MVP 范围略大**
    智能裁边、在线刮削、插件系统、Quick Look 都值得做，但不适合首版一起上。
4. **检索能力需要从 UI 功能提升到索引策略**
    支持作者、类型、年份、tag 检索，底层就必须先定义规范化字段、别名、拼音索引、全文索引，不然搜索体验会很快变得不可控。

## 三、技术架构建议

基于你的技术背景（C++/Windows/跨平台探索），推荐以下方案：

### 方案A：Avalonia + .NET（首选）
- **优势**：真正的跨平台（Win/Linux/macOS），XAML布局熟悉后类似WPF，单文件发布体积小（你关心的部署体积）
- **图像解码/渲染**：SkiaSharp 更适合作为阅读器主渲染路径；ImageSharp 更适合离线缩略图生成和服务层处理
- **压缩包**：SharpCompress 可作为首选，但建议首版只承诺 zip，rar/7z 放入增强阶段，避免把异常处理面铺太大
- **数据库**：SQLite（轻量，支持 FTS5 全文搜索）
- **异步模型**：UI线程只负责展示，扫描、缩略图、索引、网络抓取全部后台任务化

### 方案B：Qt6 + C++
- **优势**：性能极致，原生感强
- **劣势**：QML学习曲线，发布体积大，数据层和界面层需要更多自建样板代码

### 结论

如果目标是尽快做出一个跨平台且能长期演进的产品，**Avalonia + .NET + SQLite** 是更稳的起点。它不是性能上限最高的方案，但在“开发速度 / 可维护性 / 跨平台一致性”这三个维度上平衡最好。

### 关键库选择
- **缩略图生成**：不需要 ffmpeg。漫画本质是静态图像，直接从首张有效页生成封面和预览图即可
- **元数据抓取**：预留 AniList / MangaDex / Bangumi 映射接口，但首版应只做“手动编辑元数据 + 可选匹配”
- **图像缓存**：内存 LRU + 磁盘缓存，防止打开大体积 zip 时内存爆炸
- **拼音检索**：需要单独设计索引字段，不能只依赖 SQLite FTS
- **后台任务**：建议独立 `ImportWorker`、`ThumbnailWorker`、`MetadataWorker`

- **智能视图**：最近阅读（按时间）、继续阅读（未完成的）、已完结
- **筛选器**：点击标签/作者后，主界面实时过滤（非跳转新页）

补充建议：侧边栏建议固定几类高频入口：
- 全部
- 继续阅读
- 最近添加
- 最近阅读
- 未整理
- 导入失败

其中“未整理”和“导入失败”非常重要，它们决定这个软件能不能真正管理混乱资源库。

**阅读体验细节**（漫画爱好者必做）：
- **双页模式**：日系漫画从右至左，美漫从左至右，需可配置
- **页码保留**：记录具体页码，而非"第30%位置"（漫画页有固定内容）
- **缩略图导航**：底部悬停显示胶片条，快速跳转
- **智能裁边**：这是加分项，不建议进入 MVP；首版只做“适应宽度 / 适应高度 / 原始比例 / 双页拆分”即可
- **预加载**：当前页显示时，预加载前后各5页，翻页零等待

还建议补充几个漫画阅读刚需：
- **双页封面单独显示**：卷首封面通常应单页显示，避免和正文错误拼接
- **跨卷继续阅读**：当前卷读完后可直接跳下一卷
- **阅读方向独立配置**：系列级别可覆盖全局设置
- **跳过无效页**：支持过滤广告页、版权页、重复页
- **进度写回策略**：退出、翻页间隔、窗口失焦时分别如何持久化

### 添加漫画界面

**拖拽区设计**：
- 支持拖拽文件夹/ZIP/URL到任意位置
- **智能识别**：拖入`鬼灭之刃/第01卷/`时，自动识别系列关系
- **元数据匹配**：解析文件名中的卷号（Vol.01、第1卷、v1等模糊匹配）

补充建议：导入界面必须展示“将要发生什么”，例如：
- 识别为新系列 / 已有系列
- 识别为第几卷
- 是否存在重复文件
- 是否缺失封面页或有效图片

用户需要在导入前有一次确认机会，否则错误归类后续整理成本会很高。

**URL导入**：
- 不建议在主程序内直接内置爬虫逻辑，建议设计为“Provider 适配层”
- 主程序只定义统一接口：获取作品、章节列表、页面图片、登录态、限流策略
- **不要直接内置侵权源**，提供插件 API 或外部 provider 协议让用户自行对接
- URL 导入的数据应标记为远程来源，与本地 zip/文件夹分开管理

## 五、导入与索引流程设计

这一部分建议单列，因为它决定了软件是否可靠。

### 1. 本地导入状态机

建议把每一次导入视为任务，而不是一次函数调用：

1. `发现文件`：用户拖入目录、zip、多个路径
2. `结构识别`：判断是单卷、系列目录、还是混合目录
3. `页清单生成`：扫描图片文件，提取顺序、尺寸、校验信息
4. `元数据猜测`：从文件名解析系列名、卷号、作者候选
5. `冲突检测`：检测是否与已有卷重复、是否命名冲突
6. `用户确认`：必要时允许手动修正系列和卷号
7. `落库`：写入数据库
8. `后台生成`：封面、缩略图、全文索引、统计信息

只有把导入拆成任务，后续你才能自然支持中断恢复、批量导入、错误重试、导入日志。

### 2. URL 导入状态机

URL 导入建议独立于本地导入：

1. `Provider 解析 URL`
2. `获取作品元数据`
3. `获取章节列表`
4. `选择导入范围`
5. `下载页面或建立远程映射`
6. `写入本地缓存`
7. `建立书库记录`

这里有一个关键产品决策要提前定：

- **方案A：导入即下载到本地**
    优点是统一、稳定、离线可读。
    缺点是首次导入慢，占空间。
- **方案B：先建立远程书目，需要时再缓存页面**
    优点是轻量。
    缺点是实现复杂，阅读一致性差。

作为漫画阅读器首版，我建议优先 **方案A**，即“URL 导入最终仍转成本地缓存资源”。这样你的阅读器只需要面对一套页面读取抽象。

### 3. 去重策略

本地漫画库经常有重复资源，建议至少做三层去重：

1. 路径级去重：同一路径不重复导入
2. 文件指纹去重：zip 或目录内容生成哈希
3. 语义级冲突：同系列同卷号已存在时提示用户

## 六、数据模型设计

### 核心实体关系

```
LibraryItem（书库项）1 --- 1 Source（来源）
                │
                ├─ belongs to Series（可选）
                ├─ has one Volume/Book（可选卷号）
                ├─ has many Chapters（在线源可用）
                ├─ has many Pages
                ├─ has many Tags
                ├─ has many Persons（作者/画师/译者）
                └─ has ReadingProgress / ReadingHistory
```

建议采用下面这组实体，而不是只停留在 Series / Volume / Page：

### 关键实体建议

#### Series
- `id`
- `title`
- `title_sort`
- `title_pinyin`
- `alt_titles`
- `description`
- `status`（连载中 / 已完结 / 未知）
- `year`
- `cover_asset_id`
- `metadata_source`

#### Volume
- `id`
- `series_id`
- `volume_number`
- `subtitle`
- `display_name`
- `release_date`
- `page_count`
- `reading_direction_override`

#### Source
- `id`
- `kind`（folder / zip / remote）
- `path_or_url`
- `fingerprint`
- `imported_at`
- `last_scanned_at`
- `scan_status`
- `error_message`

#### Page
- `id`
- `volume_id`
- `source_id`
- `page_index`
- `inner_path`
- `width`
- `height`
- `file_size`
- `checksum`

#### Person
- `id`
- `name`
- `normalized_name`
- `role`（作者 / 画师 / 原作 / 译者）

#### Tag
- `id`
- `name`
- `category`（题材 / 来源 / 内容分级 / 自定义）
- `color`

#### ReadingProgress
- `id`
- `volume_id`
- `current_page`
- `completed`
- `last_read_at`
- `reading_mode`

这样设计的好处是：

1. 本地和在线来源能统一进同一书架
2. 作者、译者、汉化组可以分角色建模
3. 搜索和筛选字段可以做规范化，不会被展示文本绑死
4. 后续做同步、统计、历史记录时不需要推倒重来

### 文件扫描逻辑
```cpp
// 伪代码示例：智能识别系列结构
void scanPath(path) {
    if (isZip(path)) {
        // 单本：test.zip -> 解析为"test"系列，第1卷
        createVolume(path.parent, series="test", vol=1);
    }
    else if (isDirectory(path)) {
        // 检查二级目录
        if (hasSubdirs(path) && allSubdirsAreVolumes(path)) {
            // 结构：SeriesName/Volume 1/...jpg
            series = createSeries(path.name);
            for (subdir : path.subdirs) {
                volNum = extractVolumeNumber(subdir.name); // "Vol 1"->1
                createVolume(subdir, series, volNum);
            }
        }
        else {
            // 平铺文件夹：视为单本
            createVolume(path, series=path.name, vol=1);
        }
    }
}
```

这段伪代码的思路对，但建议补两个现实分支：

1. 混合目录：一个目录下同时有图片和子目录，不能简单归类
2. 噪声文件：`__MACOSX`、`.DS_Store`、`Thumbs.db`、说明文本、校验文件都要过滤

建议识别规则按优先级而不是 if/else 写死：

1. 是否为支持的压缩包
2. 是否存在足够数量的有效图片
3. 子目录是否明显符合卷命名规则
4. 根目录图片和子目录图片是否混用
5. 是否需要进入“待用户确认”状态

对漫画这种脏数据场景，**识别失败但允许用户修正**，比“自作聪明但分错类”更重要。

## 七、搜索与标签系统

**多维度检索**（这是管理大库的核心）：

1. **即时搜索**：输入即过滤，支持拼音首字母（guimie->鬼灭）
2. **高级搜索语法**：
   - `author:藤本树 tag:悬疑 status:连载` 
   - `added:>2024-01-01`（最近添加）
   - `unread`（未读过滤）
3. **标签云**：按使用频率显示，支持颜色编码（红色-热血，蓝色-治愈）
4. **智能标签**：自动提取文件名中的`[汉化组]`、`[BD]`、`[18+]`作为标签

补充建议：

### 1. 搜索字段必须分层

- **标准字段**：作者、年份、状态、卷号、是否已读
- **可归一化字段**：系列名、别名、拼音、罗马字
- **自由文本字段**：简介、备注、文件名残留信息

这样做之后，搜索体验会稳定很多。比如 `作者:藤本树` 应该走结构化过滤，而不是全文模糊匹配。

### 2. 标签需要分类型

不要把所有标签都塞到一个平面集合里，建议至少区分：

- 题材标签：热血、悬疑、校园
- 来源标签：汉化组、扫描版、电子版
- 内容标签：全年龄、R15、R18
- 用户标签：收藏、待补、重读

### 3. 首版不要过度追求“智能标签”

从文件名自动提取 `[汉化组]`、`[BD]` 有价值，但误判率会不低。建议首版把它们作为“候选标签”，由用户确认后落库。

### 4. 检索样例建议扩展

- `author:"藤本树" tag:悬疑`
- `series:"鬼灭之刃" unread:true`
- `year:>=2020 status:连载中`
- `source:remote`
- `has:duplicate`

## 八、性能与体验优化

### 缩略图策略（关键）
- **三级缓存**：内存LRU（最近20本）-> 磁盘缩略图（SQLite Blob或独立文件夹）-> 原图
- **后台生成**：导入时不阻塞UI，后台线程生成封面缩略图（200x300px WebP格式）

补充建议：缩略图更适合落独立文件夹，不建议一开始就塞 SQLite Blob。文件缓存更直观，也更方便调试、清理和迁移。

### ZIP读取优化
- **不解压到临时目录**：直接流式读取 zip 内图片
- **索引缓存**：首次打开ZIP时，记录内部文件列表到数据库，后续秒开

补充建议：页图读取应有统一抽象，例如：

- `OpenVolume()`
- `EnumeratePages()`
- `ReadPageStream(index)`

这样文件夹和 zip 可以走同一套阅读管线。

### 跨平台适配
- **Windows**：优先保证路径编码、长路径、文件锁行为正确
- **macOS**：先保证沙箱外普通文件访问稳定，Quick Look 不必首版支持
- **Linux**：遵守XDG标准，配置放`~/.config/mangareader/`

跨平台真正容易踩坑的点还有：

1. 路径大小写敏感差异
2. 文件监听行为差异
3. 非 ASCII 文件名
4. 鼠标滚轮和触控板手势差异
5. 不同平台字体 fallback 差异

## 十、避坑指南

1. **编码问题**：ZIP文件内日文/中文文件名编码混乱（Shift-JIS vs UTF-8），必须自动检测
2. **大图片崩溃**：扫描版单页可能50MB+，必须避免整卷预解码，不要一次性读入内存
3. **系列识别陷阱**：`第10卷`在字符串排序时可能在`第2卷`前面，必须智能提取数字排序
4. **封面页陷阱**：双页模式下封面和内封通常不能直接参与自动配对
5. **导入幂等性**：同一路径重复扫描时，不能生成重复记录
6. **元数据污染**：自动抓取的作者、年份、标签必须允许用户覆盖，且保留来源信息
7. **远程依赖脆弱性**：URL 导入如果依赖站点 HTML 结构，后续维护成本会非常高

## 十一、推荐的首版产品定义

如果从“资深漫画读者真的会不会长期使用”这个标准来判断，我会把首版产品定义成：

**一个以本地漫画库为核心、系列管理可靠、翻页足够快、进度足够准的跨平台阅读器。**

首版最该打磨的是这五件事：

1. 导入混乱文件时不容易分错类
2. 系列、卷号、阅读进度足够可信
3. zip 和文件夹两种来源都稳定
4. 搜索能快速找到书
5. 阅读器足够顺手，翻页没有明显等待

如果这五件事做好，这个产品即使还没有在线源和花哨特效，也已经会有明显价值。
